Development Approach — How to build the MVP?

Three strategies for getting from zero to working demo

A

Visual-First Layered (Recommended)

Phase 1: All frontend UI — terminal shell, ASCII scene frames for all states, scanline/glow effects, status panel, boot animation. Uses mock data.

Phase 2: Voice input/output — Web Speech API, mic button, TTS playback.

Phase 3: Backend — FastAPI, /api/chat with mock responses, structured JSON.

Phase 4: Integration — wire frontend to backend, localStorage persistence.

PRO: You see the full visual experience early. Matches your "UI first" priority.

CON: Integration risk is back-loaded. Real flow only works in Phase 4.

B

Vertical Slice

Sprint 1: Minimal end-to-end: boot screen → click wake → simple pet frame → mock backend → display reply. Proves the full stack works.

Sprint 2: Add voice input, TTS output.

Sprint 3: Polish — all ASCII frames, effects, status panel, localStorage.

PRO: Full stack validated early. Integration risk is lowest.

CON: The demo looks rough until Sprint 3. You stare at a basic cat for a while.

C

Parallel Component Build

Frontend and backend scaffolded simultaneously. Each component (ASCIIPet, VoiceButton, ChatAPI, PetBrain) built independently with its own mock data. Integrated at the end.

PRO: Fastest wall-clock time if everything goes right.

CON: Highest integration risk. Debugging cross-stack issues at the end is painful. Not recommended for a solo developer.
